## Problem

NH-0201 confirmed the write-path *relationship / cascade / fixup* surface is green, but did not
enumerate the CLR-type x value-range x provider WRITE fidelity matrix cell by cell. The Query/LINQ
matrix (NH-0102) surfaced the first real write-path defect while probing the integer OrderBy cell:
`ulong` values above `long.MaxValue` were SILENTLY stored as a negative two's-complement bit pattern
(silent corruption + reversed ordering + `OverflowException` on read-back). This ticket is the granular
write-path value-fidelity matrix, worked one cell at a time: probe each CLR type's write/read round-trip
and its boundary/overflow/null edges against an oracle, classify, and fix or pin+document. It stays In
Progress until every cell closes.

Acceptance criteria:

Each cell = one CLR type's write/read round-trip fidelity, including boundary and out-of-range values,
across the four providers. A cell closes only with a committed round-trip/ordering test; a divergence is
classified and fixed or pinned+documented, never left silent. `[x]` closed, `[~]` partial, `[ ]` open.

- [x] ulong -> stored as signed 64-bit (no provider has a portable native unsigned-64 type). Values in
      [0, long.MaxValue] round-trip, order, compare, and compute exactly like a long; a value above
      long.MaxValue FAILS LOUD with `NormUsageException` on every write path (direct / prepared-insert /
      compiled and the Postgres COPY, SQL Server SqlBulkCopy, MySQL bulk-load native paths) rather than
      wrapping to a negative bit pattern. Maintainer product decision: fully correct in-range, fail loud
      above. Implemented in `ParameterAssign.ToStorableInt64` + the three parameter binders + the three
      native bulk paths; pinned by `UlongStorageContractTests`; documented in docs/linq-support.md.
- [x] uint / ushort / byte / sbyte / short -> CORRECT: every boundary value round-trips exactly
      (Min/Max per type, uint above int.MaxValue, negative minima); storage dump shows the true values
      (uint.MaxValue stored 4294967295, positive in signed-64); WHERE-equality bound at each boundary
      matches exactly the written row. Pinned by `SmallIntegerRoundTripContractTests`. No code change.
- [x] decimal -> NUMERIC value round-trips exactly at full 28-digit precision (28-sig-digit value,
      decimal.MaxValue/MinValue all verbatim). The scale REPRESENTATION normalizes (trailing zeros
      trimmed, integral values gain `.0`: 2m -> 2.0m, 10.50m -> 10.5m) - DESIGN-EXCEPTION: the full
      scale-matrix probe proved the stored text is BYTE-IDENTICAL to a raw SqliteParameter binding,
      i.e. it is Microsoft.Data.Sqlite's canonical decimal text, shared by EF Core (the earlier probe
      note's "EF Core preserves scale" was a wrong assumption, falsified by the driver comparison -
      the planned product question was therefore dropped, not asked). Every C#-observable semantic is
      preserved: decimal == and GetHashCode are scale-insensitive, and Distinct/WHERE-equality merge
      scale variants exactly like LINQ-to-Objects. Pinned by `DecimalScaleRoundTripContractTests`
      (value fidelity, driver-identity, scale-variant equality).
- [x] double / float -> BIT-EXACT round-trip for every finite value probed: 17-digit values
      (0.1+0.2), smallest subnormal (double.Epsilon = 5E-324), mid-subnormal 1E-310, smallest
      normal, Max/Min - all DoubleToInt64Bits/SingleToInt32Bits identical on read-back; WHERE-equality
      at exact doubles (incl. a subnormal) matches exactly the written row. NaN fail-loud, +/-Infinity
      round-trip, and -0.0 sign loss remain pinned by `DoubleOrderingContractTests` (NH-0102 double
      cell). Pinned by `FloatingPointRoundTripContractTests`. CORRECT - no code change.
- [x] DateTime / DateTimeOffset / DateOnly / TimeOnly / TimeSpan -> ticks/values round-trip EXACTLY
      for all five types (100ns DateTime ticks, DateOnly Min/Max/leap-day, TimeOnly MaxValue
      23:59:59.9999999 + sub-ms, TimeSpan negative/multi-day/single-tick/Min/Max). DateTimeOffset
      preserves the ORIGINAL OFFSET verbatim (+05:00/-05:00/Z, not normalized). One DESIGN-EXCEPTION:
      DateTime KIND does not round-trip (Utc/Local-written values read back Unspecified; the stored
      text carries no Kind - identical to EF Core SQLite; wall-clock ticks intact; use DateTimeOffset
      for instant semantics). Pinned by `TemporalTypeRoundTripContractTests`.
- [x] Guid -> CORRECT: exact round-trip (Empty, all-ones, mixed); stores canonical LOWERCASE "D"
      text regardless of the source string's casing; WHERE-equality bound from an UPPERCASE-parsed
      Guid matches the stored row; list Contains matches across parse casings; Guid primary-key
      lookup resolves. Pinned by `GuidRoundTripContractTests`. No code change.
- [x] enum (all underlying types incl. ulong-backed) -> CORRECT: defined members (negative,
      byte 255, short Min/Max), UNDEFINED numeric values ((E)999 round-trips verbatim per .NET Enum
      semantics - never thrown/clamped), and [Flags] combos all exact; WHERE-equality matches
      undefined values. ulong-backed enums follow the ulong contract: in-range exact, above
      long.MaxValue fails loud with NormUsageException (guard verified end-to-end through the enum
      binder branch). Pinned by `EnumRoundTripContractTests`. No code change.
- [x] string / char / byte[] -> CORRECT: strings round-trip ordinally exact (BMP non-ASCII, astral
      surrogate pairs/emoji, empty, ~100k chars, leading/trailing whitespace, embedded
      quotes/newlines/tabs); WHERE-equality and Contains match quoted and astral content; chars
      exact incl. non-ASCII and U+FFFD; byte[] byte-exact incl. the EMPTY array (length 0, not null)
      and a 1MB blob. Pinned by `TextAndBlobRoundTripContractTests`. No code change.
- [x] nullable-of-any -> CORRECT: null written through nORM stores as SQL NULL (raw dump asserted)
      and reads back null for every family (int?, ulong?, decimal?, double?, DateTime?,
      DateTimeOffset?, Guid?, enum?, string?, byte[]?), while non-null values in the same columns
      round-trip exactly; WHERE col == null matches only the null row and != null only the value
      row, like C#. Pinned by `NullableColumnRoundTripContractTests`. No code change.

## Parity audit

      silent restriction: the out-of-range case throws a clear `NormUsageException` instead of corrupting.
