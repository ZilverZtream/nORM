## Problem

Confirmed by the NH-0602 temporal+migration probe: the migration layer has zero temporal
awareness. On trigger-emulated temporal storage (the default on all four providers):

1. A SQLite migration that RECREATES a temporal main table (DROP COLUMN, type change, etc.)
   silently drops the three versioning triggers. Updates after the migration write NO history
   until some future context happens to re-run the temporal bootstrap - a version is PERMANENTLY
   missing from the reconstruction chain (probed: V=1 -> V=2 -> V=4 with V=3 unrecorded).
2. On ALL trigger-emulated providers, ADD COLUMN never extends the history table and never
   regenerates the column-list triggers, so the new column's values are silently absent from
   history forever. DROP/RENAME leave history and triggers inconsistent with the main table.

SQL Server NATIVE system-versioning propagates schema changes automatically; the emulated mode
that nORM manages must do the same. Product decision (user): FULL temporal-aware migrations -
the schema snapshot marks temporal tables, every migration generator mirrors main-table schema
changes onto `<Table>_History` in lock-step (SQL Server parity: drops drop history values too),
and versioning triggers are re-emitted from the NEW schema after any main-table change.

- [x] MODEL: `TableSchema.IsTemporal` populated by `SchemaSnapshotBuilder.Build(DbContext)` when

      (from the mapping's tenant column) so regenerated triggers keep their tenant-scoped
      history-close predicate. Public API snapshot regenerated for both properties.
- [x] SQLITE: DONE - `SqliteMigrationSqlGenerator` emits temporal companion DDL for every
      IsTemporal table the diff touches, both directions: history mirrored in lock-step
      (ADD COLUMN -> history ADD COLUMN with the same DEFAULT backfilling old rows;
      DROP/type-change -> history recreate preserving surviving values; RENAME -> history RENAME;
      added/dropped TABLES create/drop the full companion set) and the three versioning triggers
      re-emitted from the direction's post-state schema (explicit DROP TRIGGER IF EXISTS first -
      the bootstrap's CREATE IF NOT EXISTS would keep stale triggers alive). Trigger DDL extracted
      to `SqliteTemporalDdl`, the single source of truth shared by the runtime bootstrap
      (`SqliteProvider.GenerateTemporalTriggersSql` now delegates) and the generator. Pinned
      END-TO-END by `TemporalMigrationContractTests` (7 tests): versioning continues IMMEDIATELY
      after Up with NO missing version (the exact silent-stop the probe found), history lock-step
      on add/drop/rename, added-column values in history from the first post-migration write,
      AsOf spanning the migration on both models, Down restoring shape + versioning, and
      non-temporal tables receiving zero temporal DDL.
- [x] SQLSERVER/POSTGRES/MYSQL emulated: DONE at the SQL-text level - all three server generators
      emit temporal companion DDL for IsTemporal tables in both directions: history mirrored in
      lock-step (ADD carries the same DEFAULT for backfill parity; DROP drops history values;
      RENAME follows - sp_rename on SQL Server; ALTERED columns mirror each generator's own
      strategy, DROP+ADD on Postgres/MySQL and ALTER COLUMN on SQL Server; added/dropped TABLES
      create/drop the full companion set, incl. dropping PostgreSQL's orphaned trigger function),
      and the versioning triggers re-emitted from the direction's post-state schema on any
      column-set change (server ALTERs do not drop triggers, but the column-list bodies go
      stale). Per-provider trigger DDL extracted to `SqlServerTemporalDdl` /
      `PostgresTemporalDdl` / `MySqlTemporalDdl` - single sources of truth shared by each
      runtime bootstrap (provider methods now delegate) and generator. Shared per-direction
      column-set reconstruction in `TemporalMigrationSchema` (carries IsTemporal +
      TenantColumnName). Pinned by `ServerTemporalMigrationDdlContractTests` (15 tests across
      the three generators: mirror + regen on add-column, tenant predicate preserved, added/
      dropped table companion lifecycle, zero temporal DDL for non-temporal tables).
      Behavioural verification on live servers is the live-pending cell.
- [x] DOWN direction: covered in the same implementations and pinned alongside Up.
      SQLite behavioural: `Down_restores_history_shape_and_versioning_continues` (history regains
      the dropped column, triggers regenerate from the restored schema, the next write is
      versioned) plus the added/dropped-table Down lifecycle tests. Servers (SQL-text):
      `Added_column_mirrors_onto_history_and_regenerates_triggers` asserts the Down history
