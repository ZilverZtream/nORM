## Problem

Confirmed by the NH-0602 temporal+migration probe: the migration layer has zero temporal
awareness. On trigger-emulated temporal storage (the default on all four providers):

1. A SQLite migration that RECREATES a temporal main table (DROP COLUMN, type change, etc.)
   silently drops the three versioning triggers. Updates after the migration write NO history
   until some future context happens to re-run the temporal bootstrap - a version is PERMANENTLY
   missing from the reconstruction chain (probed: V=1 -> V=2 -> V=4 with V=3 unrecorded).
2. On ALL trigger-emulated providers, ADD COLUMN never extends the history table and never
   regenerates the column-list triggers, so the new column's values are silently absent from
   history forever. DROP/RENAME leave history and triggers inconsistent with the main table.

SQL Server NATIVE system-versioning propagates schema changes automatically; the emulated mode
that nORM manages must do the same. Product decision (user): FULL temporal-aware migrations -
the schema snapshot marks temporal tables, every migration generator mirrors main-table schema
changes onto `<Table>_History` in lock-step (SQL Server parity: drops drop history values too),
and versioning triggers are re-emitted from the NEW schema after any main-table change.

Acceptance criteria:

`[x]` closed, `[~]` partial/live-pending, `[ ]` open.

- [ ] MODEL: `TableSchema.IsTemporal` populated by `SchemaSnapshotBuilder.Build(DbContext)` when
      the context has temporal versioning enabled in an emulated storage mode (provider-native
      storage is provider-managed and stays unflagged). Assembly-based Build cannot see context
      options - documented in its existing differences list.
- [ ] SQLITE: recreate/ADD/DROP/RENAME on a temporal table mirrors the change onto
      `<Table>_History` in lock-step and re-emits the three versioning triggers from the NEW
      schema inside the same migration script. Contract: versioning continues IMMEDIATELY after
      the migration (no bootstrap needed, no missing version), AsOf spans the migration, and an
      added column's values appear in history from the first post-migration write.
- [ ] SQLSERVER/POSTGRES/MYSQL emulated: the same lock-step mirror + trigger re-emission in their
      generators (SQL-text level pinned from SQLite-side tests; behavioural verification on live
      servers is the live-pending cell).
- [ ] DOWN direction: reverting a migration on a temporal table also mirrors history and
      regenerates triggers from the reverted schema.
- [ ] LIVE-PENDING: behavioural verification on live SqlServer/Postgres/MySQL, plus the
      provider-native SqlServer interaction (native mode must NOT receive emulated mirror DDL).
