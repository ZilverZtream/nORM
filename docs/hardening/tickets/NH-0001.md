# NH-0001 - hardening ticket gate and no-overclaim process

Ticket: NH-0001
Status: Verified
Owner: unassigned
Area: process/gate
Priority: P0
Quarantine allowed: no
Named gate: scripts/validate_hardening_ticket.ps1 -Ticket NH-0001

## Problem

Broad checklist items (e.g. `docs/v1-sharpening/`) let an agent implement the smallest visible
fix, mark a box done, and leave the larger category broken. nORM needs a gate that ties every
hardening change to an accepted ticket with explicit scope, acceptance criteria, evidence, and
reference-app impact - enforced by tooling, not merely described. Until enforcement exists, the
hardening docs are honour-system and can be overclaimed.

## Non-negotiable scope

This ticket is not complete until the process is enforced, not merely described.

Allowed paths:

- docs/hardening/**
- scripts/validate_hardening_ticket.ps1
- scripts/install_hardening_hooks.ps1
- .githooks/**
- .github/**

Acceptance criteria:

- [x] `docs/hardening/README.md` defines the status vocabulary and closure (no-overclaim) rules.
- [x] `docs/hardening/ticket-template.md` defines the required ticket fields.
- [x] `scripts/validate_hardening_ticket.ps1` rejects a missing / Draft / malformed ticket id.
- [x] The script rejects changed files outside the ticket's `Allowed paths`.
- [x] The script allows completed `[x]` criteria on `In Progress` tickets but rejects
      `Status: Verified` while any acceptance criterion is `[ ]` or `[~]`.
- [x] The script rejects new restriction-language (`unsupported`/`not implemented`/`TODO`/
      `deferred`/`rejected`/`missing`/`non-goal`/`limitation`) added to source outside policy
      docs or negative tests unless the owning ticket carries a Parity-audit classification.
- [x] A pre-commit hook (`.githooks/pre-commit`) invokes the script for hardening changes;
      `scripts/install_hardening_hooks.ps1` wires it up.
- [x] The developer/agent workflow (`docs/hardening/README.md`) documents passing `-Ticket NH-NNNN`.

## Parity audit

- [x] INTERNAL-INVARIANT: this is a process ticket; it introduces no product restriction. It
      only forces later feature tickets to classify restrictions and attempt implementation
      first.

## Correctness-first audit

- [x] IMPLEMENTATION-FIRST: NH-0001 implements the guard that forces later feature tickets to
      attempt a portable nORM implementation (differential/oracle-proven) before accepting a
      `NormUnsupportedFeatureException` throw-pin or a docs-only caveat.
- [x] The validator itself is demonstrated with failing runs (below) before this ticket is
      marked `Verified`.

## Required verification

- `powershell -NoProfile -ExecutionPolicy Bypass -File scripts/validate_hardening_ticket.ps1 -Ticket NH-0001`
- Demonstrate one failing run with an invalid / Draft ticket id.
- Demonstrate one failing run with a staged path outside `Allowed paths`.
- Demonstrate one failing run setting `Status: Verified` while an acceptance criterion is `[ ]`.
- Demonstrate one failing run adding an unclassified restriction-language hit in source.

## Reference app impact

None directly. This ticket makes the *process* enforceable; feature tickets make
`samples/nORM.Sample.Store` and the live provider suite pass.

## Verification evidence

Validator demonstrated this iteration (probe files created and removed immediately):

- `-Ticket NH-0001 -NoDiff` -> exit 0 (`structure valid`).
- `-Ticket BADID -NoDiff` -> exit 1 (malformed id).
- `-Ticket NH-9999 -NoDiff` -> exit 1 (missing ticket file).
- `-Ticket NH-9998 -NoDiff` (Status: Verified with one open criterion) -> exit 1
  (`is Status: Verified but has 1 open/partial acceptance criteria (no overclaim)`).
- Staged out-of-allowed-path file, `-Ticket NH-0001 -StagedOnly -IgnoreUntracked` -> exit 1
  (`Changed files outside NH-0001 Allowed paths: PROBE_OUTSIDE.txt`).
- Staged `src/` restriction-language under a ticket with no Parity-audit class,
  `-Ticket NH-9997 -StagedOnly -IgnoreUntracked` -> exit 1
  (`Added restriction-language in source without a Parity-audit classification`).
- Hook + installer: `.githooks/pre-commit` runs the validator when `NORM_HARDENING_TICKET`
  is set; `scripts/install_hardening_hooks.ps1` sets `core.hooksPath=.githooks`. This ticket's
  own commit is gated through that hook.
