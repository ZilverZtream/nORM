## Problem

Cache correctness is adversarial: a cacheable query must be invalidated by a write to ANY table
it reads (including correlated subqueries), or it serves a silent stale read. The targeted
invalidation tests (`CorrelatedSubqueryCacheInvalidation`, `MultiTableCacheInvalidation`,
`CacheInvalidationAcrossNavigation`) cover fixed shapes, but the cache-tag bug history shows
staleness can hide from happy-path probes (a multi-table tag bug survived ~10 passing probes).
A differential fuzzer over randomized shapes and writes is needed.

Acceptance criteria:

- [x] A seeded differential cache-staleness fuzzer exists: per seed it builds a random
      Parent/Child dataset, warms a random Cacheable query shape, applies a random write to a
      table that shape reads, then asserts the cached context's re-run equals a fresh (uncached)
      context computed against the same shared database (the oracle).
- [x] The fuzzer has teeth: routing the write through a non-invalidating context (a deliberate
      sabotage) makes it fail, catching a real stale read - demonstrated, then reverted.
- [x] The committed fuzzer runs dry over its seed sweep (120 seeds).
- [x] Shapes covered: parent-scalar (reads Parent only), correlated child COUNT for one parent,
      and correlated child SUM for all parents (reads Parent + Child); writes covered: tracked
      insert (parent/child), bulk `ExecuteDeleteAsync` (parent/child), and direct `UpdateAsync`
      (child).

## Parity audit

- [x] IMPLEMENTATION-DEBT: the fuzzer's shapes are correlated-subquery + parent-scalar; explicit
      multi-table JOIN projection staleness is covered by the targeted `MultiTableCacheInvalidation`
      tests, not yet by this fuzzer. Extending it to join and window-function shapes is real
      remaining work (stays open); it does not block this ticket's scoped deliverable.

## Correctness-first audit

- [x] Differential proof, not happy-path: the oracle is a fresh uncached context reading the same
      DB; a missed invalidation diverges from it.
- [x] Teeth demonstrated before claiming green (the sabotage run failed at seed 0).
- [ ] EF Core / Dapper parity: not applicable (nORM-specific cache invalidation).
