## Problem

nORM exposes both a direct-write API (`InsertAsync`/`UpdateAsync`/`DeleteAsync`, immediate) and
change tracking (`Add` + `SaveChangesAsync`), plus bulk and set-based paths. Having all three is
a strength, but "when do I use which" was undocumented, which is a real adoption friction and the
historical source of direct-op / tracker interaction confusion. A decision-oriented guide was
missing (the mechanics doc `docs/change-tracking.md` covers tracking internals, not the choice).
