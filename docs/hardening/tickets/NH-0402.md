## Problem

The migration up/down oracle fuzzer (MigrationDataPreservationFuzz) and the recreate-path fixes
established the broad contracts (one recreate per table per direction; dropped-NOT-NULL Down always
recreates; type-zero backfill). This ticket is the granular per-cell matrix on SQLite, worked with
the differential method: apply a single schema operation to a table seeded with boundary values of a
specific type family, verify the DATA survives byte-exactly, then apply the Down migration and
verify schema AND data round-trip. Each cell isolates one operation x one type family so a
data-eating recreate or a backfill regression is attributable to a specific pair.

Acceptance criteria:

Each cell = one migration operation applied over seeded boundary data, up-verified and down-verified.
`[x]` closed, `[~]` partial, `[ ]` open.

- [x] ADD nullable column x type family -> CORRECT: with boundary values seeded in stored form
      (17-digit decimal text, offset-suffixed DTO, canonical Guid, astral+quoted string, blob,
      in-range ulong as signed-64), the Up leaves every stored byte untouched and the new column
      reads NULL; the Down (DROP via SQLite full-table recreate) rebuilds the table with the same
      rows byte-for-byte and removes the column. Pinned by
      `MigrationBoundaryDataPreservationTests`. No code change.
- [x] ADD NOT NULL column -> the ACTUAL contract is better than the criterion assumed: WITHOUT a
      DefaultValue the generator FAILS LOUD at SQL-generation time ("Cannot generate ADD COLUMN ...
      NOT NULL ... without a DefaultValue") - never a silent implicit backfill nor a mid-migration
      runtime constraint failure. WITH an explicit DefaultValue, existing rows are backfilled with
      the default, prior columns stay byte-exact, and the Down drops the column with data intact.
      (The type-zero backfill contract from the fuzzer campaign applies to the Down-RECREATE of a
      dropped NOT NULL column, a different scenario.) Pinned by both fail-loud and with-default
      tests in `MigrationBoundaryDataPreservationTests`.
- [x] DROP column -> CORRECT: the keeper columns survive the SQLite full-table recreate byte-exact
      (boundary values incl. blob/astral string/17-digit decimal); the Down restores the column with
      the documented contract (dropped-column data is gone; the re-added nullable column reads NULL)
      and the keepers stay byte-exact through the second recreate. Pinned by
      `MigrationBoundaryDataPreservationTests`. No code change.
- [x] RENAME column -> CORRECT: ColumnSchema.PreviousName drives the rename through the recreate;
      the renamed column carries S's exact bytes under the new name (all sibling columns byte-exact),
      and the Down renames back with the original snapshot matching byte-for-byte. Pinned in
      `MigrationBoundaryDataPreservationTests`. No code change.
- [x] up-then-down FULL round-trip -> CORRECT: one diff combining ADD + RENAME + DROP over the
      boundary table, applied Up then Down, restores the schema identically (pragma_table_info
      names/types/notnull/pk string-compared) and every keeper column byte-exact; the dropped
      column returns NULL per the documented contract. Pinned in
      `MigrationBoundaryDataPreservationTests`. No code change.
- [x] index / unique-constraint add + drop -> CORRECT: adding a UNIQUE index over boundary data
      leaves every byte untouched, the constraint is ENFORCED (duplicate insert throws
      SqliteException), and the Down removes the index (pragma_index_list empty) with data intact.
      Pinned in `MigrationBoundaryDataPreservationTests`. No code change.
- [x] FK add / drop -> CORRECT: adding an FK from a seeded child to MigBoundary (child recreate under
      PRAGMA foreign_keys=ON) leaves parent data untouched and ENFORCES the constraint (orphan insert
      throws SqliteException); after the Down the same orphan insert is allowed and data is intact.
      Pinned in `MigrationBoundaryDataPreservationTests`. No code change.
- [x] DEFAULT-value column add -> IMPLEMENTATION-DEBT FOUND AND FIXED: a NULLABLE added column's
      declared DefaultValue was SILENTLY DROPPED by the SQLite, MySQL, and Postgres generators (only
      `NULL` emitted, no DEFAULT clause) - the migrated schema diverged from a freshly-created one
      (CREATE TABLE and the Down-restore both emit the default) and new inserts did not honour the
      model's default; SqlServer was already correct (unconditional inline default clause). Probe:
      existing row Extra=42 count was 0. Fixed in all three generators; now existing rows are
      backfilled with the default, new inserts honour it, and the Down drops the column with data
      intact. (The computed-to-plain transition re-add sites share the pattern and are NOTED for the
      computed-column cell - not fixed blind without coverage.) Pinned in
      `MigrationBoundaryDataPreservationTests`; full Category=Fast green (11033).
