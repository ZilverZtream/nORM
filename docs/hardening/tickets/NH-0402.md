## Problem

The migration up/down oracle fuzzer (MigrationDataPreservationFuzz) and the recreate-path fixes
established the broad contracts (one recreate per table per direction; dropped-NOT-NULL Down always
recreates; type-zero backfill). This ticket is the granular per-cell matrix on SQLite, worked with
the differential method: apply a single schema operation to a table seeded with boundary values of a
specific type family, verify the DATA survives byte-exactly, then apply the Down migration and
verify schema AND data round-trip. Each cell isolates one operation x one type family so a
data-eating recreate or a backfill regression is attributable to a specific pair.

Acceptance criteria:

Each cell = one migration operation applied over seeded boundary data, up-verified and down-verified.
`[x]` closed, `[~]` partial, `[ ]` open.

- [ ] ADD nullable column x type family -> existing rows keep byte-exact values; new column NULL.
- [ ] ADD NOT NULL column (type-zero backfill contract) -> existing rows backfilled with the type
      zero; prior columns byte-exact.
- [ ] DROP column -> remaining columns byte-exact through the SQLite recreate; Down restores the
      column with the documented backfill contract.
- [ ] RENAME column -> data byte-exact under the new name; Down renames back with data intact.
- [ ] up-then-down FULL round-trip -> schema (columns, types, nullability, PKs) and data identical
      to the pre-migration state for a multi-type table (decimal/temporal/Guid/blob columns).
- [ ] index / unique-constraint add + drop -> data untouched; constraint enforced after add
      (duplicate insert fails loud), gone after drop.
- [ ] FK add / drop -> data untouched; violating insert fails loud after add.
- [ ] DEFAULT-value column add -> existing rows get the default; new inserts honour it.
