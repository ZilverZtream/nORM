## Problem

Result-cache correctness is adversarial: the multi-table tag bug (correlated-subquery tables
missing from plan.Tables -> silent stale read) survived ~10 happy-path probes, and the closure
plan-cache bug baked captured values into cached plans. NH-0801's staleness fuzzer covers random
shape x write against an uncached oracle, and the tenant matrix pinned tenant/DB-identity keying.
What no suite pins per-cell: TRANSACTION interplay (a Cacheable read inside a transaction that
later ROLLS BACK must not leave a poisoned entry serving never-committed data), per-write-ROUTE
invalidation (each write API individually), expiration-parameter contracts, and non-tenant
cross-context sharing. This matrix closes those cells differentially.

- [x] TRANSACTION x cache -> IMPLEMENTATION-DEBT FOUND AND FIXED: a Cacheable read inside an
      explicit transaction cached the transaction's UNCOMMITTED state; after ROLLBACK the cache
      kept serving the never-committed row to later readers (probe: raw truth 1 row, cached
      answer 2 rows - silent phantom data). Fix: the result cache participates only OUTSIDE
      transactions (`ResultCacheUsable` gates all six cache entry points on
      `plan.IsCacheable && CacheProvider != null && CurrentTransaction == null && ambient
      Transaction.Current == null`) - an in-transaction cache HIT would equally hide the
      transaction's own writes, so both directions are gated. The commit direction was already
      correct (SaveChanges invalidates tags). Pinned by
      `TransactionCacheInteractionContractTests` (rollback-poisoning + commit-invalidation).
- [x] WRITE-ROUTE INVALIDATION -> CORRECT and now PINNED by `CacheWriteRouteInvalidationContractTests` (direct + batched SaveChanges, BulkInsert/Update/Delete, ExecuteUpdate/ExecuteDelete each individually re-serve fresh data through an UNTRACKED cached read). PROBE LESSON recorded: the first probe asserted through TRACKED reads and looked like two invalidation kills - the stale values were the identity map returning already-tracked instances after set-based writes, which is EF-IDENTICAL semantics, not a cache defect. Raw ADO SQL bypasses nORM and cannot invalidate (see Parity audit). Original scope: a cached result is invalidated by EVERY write route
      individually: SaveChanges (direct + batched), BulkInsertAsync/BulkUpdateAsync/
      BulkDeleteAsync, ExecuteUpdateAsync/ExecuteDeleteAsync. Raw ADO SQL bypasses nORM and
      cannot invalidate - classify that boundary honestly. Extends the caching.md open item
      "bulk-write invalidation covers all touched tables".
- [x] EXPIRATION CONTRACTS -> CORRECT and now PINNED (`CacheExpirationAndScopeContractTests`): an expired entry re-fetches fresh data past its TTL (probed by mutating behind the cache with raw SQL). TimeSpan.Zero/negative deferred to the bounded-cache cell with docs/cache-policy.md. Original scope: an expired entry re-fetches fresh data; pin the semantics of
      TimeSpan.Zero and negative expirations (immediate-expiry or reject - whatever the code
      does, decided deliberately and pinned).
- [x] CROSS-CONTEXT SHARING -> IMPLEMENTATION-DEBT FOUND AND FIXED: two SEPARATE ':memory:' connections have IDENTICAL connection strings but are DIFFERENT databases - the cache key derived database identity from the connection string alone, so a shared cache provider served database A's rows to database B (probe: B expected 3 rows, was served A's 1). Fix: new provider hook `IsConnectionScopedDatabase` (SQLite: ':memory:'/Mode=Memory without shared cache) makes the cache key include a per-connection-instance id via ConditionalWeakTable. Same-database sharing (two contexts, one connection) pinned as a HIT in the same class. Original scope: two contexts on the SAME database + shared provider
      serve each other's entries (hit); contexts on DIFFERENT databases never share (DB identity
      in the key - the tenant matrix pinned the tenant dimension; pin the plain-DB dimension).
- [x] LIVE-DATABASE-SWAP IDENTITY -> IMPLEMENTATION-DEBT FOUND AND FIXED (post-Verified adversarial
      neighbour probe, 2026-07-16): the cache key derived database identity from
      `Connection.ConnectionString`, which goes STALE after `ctx.Connection.ChangeDatabase(...)` -
      providers update `Connection.Database` but keep the original string, so a cacheable read
      after the swap served the PREVIOUS database's rows (live probe: warmed in the application
      database, swapped to master where the table does not even exist, cached row still returned).
      Reachable in default/compatibility mobility mode; strict mode's `Connection` getter throws
      before the swap. Fix: `BuildCacheKeyFromPlan` also keys on the LIVE
      `RawConnection.Database` name (constant "main" on SQLite, where the per-connection id
      already covers connection-private databases). Pinned by `CacheDatabaseSwapContractTests`
      (live SqlServer, probe-before-fix verified: unfixed code served the stale row, fixed code
      executes against the new database and fails loudly). PROBE LESSON: the first probe run was a
      FALSE POSITIVE - the session's process-level NORM_TEST_* still pointed at master, so
      "before" and "after" were the same database; live probes must print/assert the live
      `Connection.Database` and refresh process-scope env from User scope in long-lived sessions.
- [x] BOUNDED CACHE + TTL-PARAMETER CONTRACTS -> `NormMemoryCacheProvider` is bounded at 10,240
      entries (SizeLimit), matching docs/cache-policy.md verbatim; bound/eviction behaviour under
      adversarial churn is genuinely covered by the release-gate suites the policy doc names
      (`CacheMemoryBoundReleaseGateTests`, `CacheFaultInjectionStressTests`,
      `ConcurrentLruCacheStressTests`, `CacheLockConcurrencyTests`) - cited, not duplicated.
      Non-positive Cacheable TTLs: CORRECT-REJECT - `Cacheable()` validates eagerly at the call
      site with a clear message (pinned in `CacheExpirationAndScopeContractTests`); the cache
      execution paths additionally treat a non-positive resolved expiration as
      execute-uncached (defense in depth for the options-level expiration path).
