## Problem

NH-0202 verified value fidelity through the DIRECT write path (InsertAsync / prepared inserts /
parameter binders). The BULK paths - BulkInsertAsync, bulk update/delete (ExecuteUpdate /
ExecuteDelete), and the provider staging routes - bind values through different code
(EntityDataReader, DataTable staging, COPY writers, multi-row VALUES batching), and history shows
they diverge: SQLite bulk paths once bypassed value converters (silent corruption, fixed and pinned
in the bulk-converter campaign), and MySQL bulk staging silently ROUNDED sub-second temporal values
through DATETIME(0) staging tables (live-proven, fixed). This ticket is the granular bulk-path
value-fidelity matrix on SQLite, worked one cell at a time with the same differential round-trip
method as NH-0202: bulk-write crafted values per type family, dump storage, read back, compare to
what was written, and compare against the direct-path storage form (bulk and direct must store
IDENTICALLY). Provider-specific staging cells (MySQL DATETIME(6), SqlServer staging types) are noted
for live-provider runs and are NOT closable from SQLite-only evidence.

Acceptance criteria:

Each cell = one type family bulk-written (BulkInsertAsync; plus ExecuteUpdate value-binding where the
type can be SET) with boundary values, verified by storage dump + read-back + identity with the
direct-path storage form. `[x]` closed, `[~]` partial, `[ ]` open.

- [x] integers -> CORRECT through bulk: all boundary values (Min/Max per type, uint above int.Max,
      long.Min/Max) bulk-insert with storage BYTE-IDENTICAL to the direct path (sibling-table join
      matched 3/3) and read back exactly. ulong in-range (9e18) exact through bulk; ulong.MaxValue
      inside a batch FAILS LOUD (NormUsageException surfaced as NormException through the bulk
      pipeline, message intact) with ZERO rows persisted from the failed batch - atomic. Pinned by
      `BulkIntegerFidelityContractTests`. No code change.
- [ ] decimal through bulk: canonical text identical to direct path; 17th-digit values distinct;
      28-digit/Max/Min exact.
- [ ] double/float through bulk: bit-exact incl. subnormals and Max/Min; NaN fails loud like direct.
- [ ] temporal (DateTime/DTO/DateOnly/TimeOnly/TimeSpan) through bulk: ticks exact, DTO offset
      preserved, storage text identical to direct path.
- [ ] Guid through bulk: canonical lowercase text identical to direct path.
- [ ] enum (incl. undefined values, [Flags], ulong-backed guard) through bulk.
- [ ] string/char/byte[] through bulk: astral pairs, 100k strings, 1MB blobs, empty array.
- [ ] nullable-of-any nulls through bulk: SQL NULL stored, null read back.
- [ ] converter columns through bulk: value converters MUST be applied on every bulk route
      (regression-pin the bulk-converter contract: NegatingConverter + enum-to-name converter through
      BulkInsertAsync and bulk update SET values; a bypassed converter is silent corruption).
- [ ] PROVIDER-SPECIFIC (live runs required; not closable from SQLite): MySQL staging DATETIME(6)/
      TIME(6)/TINYINT UNSIGNED fidelity; SqlServer SqlBulkCopy staging types; Postgres COPY binary
      types. Recorded here; verified when the live-provider suite runs.
