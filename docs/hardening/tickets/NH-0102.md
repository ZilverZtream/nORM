## Problem

NH-0101 confirmed the *existing* LINQ suite is green but did NOT enumerate the operator x
operand-type x edge-case matrix cell by cell. The very first cell probed differentially - string
`OrderBy` vs a .NET oracle - immediately surfaced a real, previously-undocumented divergence
(SQLite ordinal collation vs .NET CurrentCulture). This ticket is the granular Query/LINQ matrix,
worked one cell at a time: probe differentially, classify (IMPLEMENTATION-DEBT / DESIGN-EXCEPTION /
CORRECT-REJECT), fix or pin+document, mark the cell. It stays In Progress until every cell closes.

Acceptance criteria:

Each cell = one operator x one operand type x its edge cases. A cell closes only with a committed
differential test vs a LINQ-to-Objects oracle; a divergence is classified and fixed or
pinned+documented, never left silent. `[x]` closed, `[~]` partial, `[ ]` open.

OrderBy / ThenBy x operand type:

- [x] string -> follows DATABASE COLLATION, not .NET CurrentCulture (SQLite BINARY/ordinal).
      DESIGN-EXCEPTION (identical to EF Core). Pinned by `StringOrderingContractTests`, documented
      in `docs/linq-support.md`.
- [x] Guid -> CONSISTENT with .NET `Guid.CompareTo` on SQLite. nORM stores the canonical lowercase
      hex string; SQLite BINARY-orders that text ordinally, which equals field-wise unsigned order
      (what `Guid.CompareTo` does) for every Guid. CORRECT - no code change; pinned by
      `GuidOrderingContractTests` with adversarial high-bit values (a BLOB/byte storage would diverge).
      SQL Server `uniqueidentifier` field-order is a provider-native divergence deferred to NH-0502.
- [ ] nullable-of-any -> NULLS FIRST/LAST differs by provider; confirm nORM's normalization.
- [ ] int / long / short / byte (signed + unsigned).
- [ ] decimal -> precision/scale boundary ordering.
- [x] double / float -> finite values and +/-Infinity order CONSISTENTLY with .NET (SQLite REAL,
      numeric ordering). Two DESIGN-EXCEPTIONs pinned (both identical to EF Core, same Microsoft.Data.Sqlite
      layer): NaN is not representable in SQLite REAL -> insert FAILS LOUD with
      `InvalidOperationException("Cannot store 'NaN' values.")` (no silent NULL), so NaN never enters an
      ordered set; -0.0 and +0.0 compare EQUAL in SQL and -0.0 loses its sign on read-back, whereas
      .NET `double.CompareTo` orders -0 < +0 (use a tiebreak for deterministic zero placement). Pinned by
      `DoubleOrderingContractTests`.
- [ ] DateTime / DateTimeOffset -> instant vs local ordering.
- [ ] TimeSpan / bool / enum / char.
- [ ] converter-mapped column -> ordering is on the provider value; prove it matches the .NET key.

## Parity audit

- [x] DESIGN-EXCEPTION: string ordering follows DB collation (matches EF Core). Not a defect;
      pinned + documented. This ticket introduces no new restriction.
- [x] CORRECT: Guid ordering on SQLite equals .NET `Guid.CompareTo` (canonical-hex TEXT storage +
      unsigned leading-field compare). Proven by diagnostic dump (storage=text, raw-SQL/nORM/.NET all
      `3,1,2,5,6,4`); pinned. SQL Server `uniqueidentifier` ordering divergence recorded for NH-0502.
- [x] CORRECT + DESIGN-EXCEPTION: double/float finite + +/-Infinity ordering matches .NET; NaN
      fails loud on store (SQLite REAL limit, matches EF Core - not silent corruption); -0.0 == +0.0 in
      SQL with sign lost on read-back. Proven by diagnostic dump; pinned. NaN/-0 write fidelity also
      recorded for the write-path matrix NH-0202.
