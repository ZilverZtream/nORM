## [CRITICAL] src/nORM/Core/ChangeTracker.cs:614
**?** — ReindexAfterInsert() mutates the shared _entriesByKey identity map without acquiring _trackLock, unlike every other mutator (Track/Remove/RemoveUnlocked), allowing it to race with a concurrent Remove() on the same entity and resurrect a stale, detached EntityEntry into the identity map.

_Failure:_ SaveChanges assigns a DB-generated key to entity A and calls ReindexAfterInsert(A, map): it reads _entriesByReference[A] (entry still present) and computes pk=42, then is preempted before calling typeDict.TryAdd(42, entry). Concurrently another thread calls ctx.Remove(A) (e.g. from a background task or any code that doesn't respect the documented 'single-threaded DbContext' contract, exactly the 'unexpected re-entry' scenario Track()/Remove() explicitly lock against): Remove() takes _trackLock, removes A from _entriesByReference, calls entry.DetachEntity() (sets entry.State=Detached and entry.Entity=null), and finds no _entriesByKey[42] entry yet to remove (no-op) — then releases the lock. ReindexAfterInsert then resumes and calls typeDict.TryAdd(42, entry), inserting the now-detached entry (Entity==null) into _entriesByKey keyed by 42. The very next query/navigation load that materializes a row with PK=42 calls ChangeTracker.Track(freshEntity, Unchanged, mapping); Track() finds the stale pkEntry via the PK map, sees pkEntry.State==Detached (not Added/Modified/Deleted) so overwrites pkEntry.State to Unchanged, and returns pkEntry instead of creating a fresh entry — but pkEntry.Entity is still null. Callers that read entry.Entity (materializer/navigation loaders, see QueryExecutor.cs:517, IncludeProcessor.cs:226/567) then hand back null or throw NullReferenceException instead of the freshly queried row, i.e. a silently wrong/empty result for a row that legitimately exists in the database.

## [CRITICAL] src/nORM/Core/DbContext.PreparedStatements.cs:232
**?** — PreparedInsertCommand.ExecuteAsync mutates a single shared DbCommand/DbParameter set with no synchronization, so concurrent InsertAsync calls on one DbContext for the same entity type can silently swap row data between entities instead of failing loudly.

_Failure:_ DbContext.InsertAsync<T>() (DbContext.WriteOperations.cs lines 37-49) takes the 'fast path' whenever there is no ambient transaction, no TenantProvider and no RetryPolicy configured (the common default setup) and reuses one PreparedInsertCommand per entity type from _preparedInsertCache (DbContext.cs line 127). PreparedInsertCommand.ExecuteAsync (lines 232-261) binds the entity's values onto the command's shared DbParameter objects, then calls ExecuteNonQueryWithInterceptionAsync/ExecuteScalarWithInterceptionAsync - all without any lock. If application code does `await Task.WhenAll(ctx.InsertAsync(orderA), ctx.InsertAsync(orderB))` on the same DbContext (a mistake users commonly make, and nothing here throws to catch it), the parameter-bind step of the second call can race ahead of the first call's already-issued ExecuteNonQuery/ExecuteScalar send, so the row actually written for orderA's statement contains orderB's field values (or vice versa). Each C# entity's generated key is still assigned via SetPrimaryKey(entity, newId) from its own awaited call, so orderA.Id looks populated and 'successful' even though the database row for that id now holds orderB's data - orderA's data is silently lost with no exception raised. The codebase already recognizes this exact hazard for cached prepared read commands: FastPathPreparedCommand (same file, line 375) has a `SemaphoreSlim Gate` that callers explicitly await/release in FastPathQueryExecutor.Simple.cs and FastPathQueryExecutor.Paging.cs, but no equivalent guard exists for PreparedInsertCommand, which is reachable via ordinary InsertAsync usage, not just the explicit PrepareInsertAsync API.

## Fix-scope note: bulk decimal truncation is deeper than the DECIMAL(18,2) literal
Read-through (gate-quiet) found the real shape:
- `SqlServerProvider.Bulk.cs` / `MySqlProvider.Bulk.cs` build staging temp-table columns via `GetSqlType(Type)` (SqlServer.Bulk.cs:310, called at :95 and :212) — it takes only a `Type`, so it has no precision/scale.
- `HasPrecision(p, s)` stores precision in the model config's precision map (keyed by PropertyInfo) explicitly "for migration snapshots"; the runtime `Column` mapping (Column.cs) carries NO precision/scale.
- Therefore the fix must PROPAGATE configured precision/scale to the bulk staging DDL (either onto the Column at model-build time, or by looking it up from the context's model config by PropertyInfo in the bulk path), and emit `DECIMAL(p, s)` for decimal staging columns.
- Unconfigured default: widen from `DECIMAL(18,2)` to a documented safe value. Caveat to document: no single SQL DECIMAL type holds every .NET decimal (max 28–29 significant digits, scale up to 28), so an unconfigured extreme-scale decimal has an inherent limit — configuring `HasPrecision` (and matching the destination column) is the correct path and is what the pin `BulkUpdateDecimalPrecisionParityTests` exercises.
