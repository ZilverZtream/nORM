# LINQ Support Matrix

nORM v1 does not claim unlimited LINQ translation. The supported contract is:
server-side translation for the query shapes below, explicit client-evaluation
rules for projections, and deterministic exceptions for shapes outside the
matrix.

Every row applies identically to both query entry points: `ctx.Query<T>()` and
its EF-parity alias `ctx.Set<T>()` return the same queryable, so examples
written as `ctx.Query<X>()` (including correlated-subquery sources) work
unchanged with `ctx.Set<X>()`.

Status values:

- Supported: intended v1 behavior for SQL Server, SQLite, PostgreSQL, and MySQL.
- Constrained: supported only under the restrictions listed.
- Preview: usable but not part of the v1 compatibility promise.
- Unsupported: must fail deterministically rather than silently changing
  semantics.

## Query operators

| Feature | Status | Notes |
| --- | --- | --- |
| `Where` predicates | Supported | Member comparisons, boolean logic, null checks, `Contains`/`StartsWith`/`EndsWith` (string **and char** needles, including `Contains(char, StringComparison)` — char needles share the string handlers, so ordinal matching, negated-NULL rescue, closure re-binding, and LIKE-metacharacter escaping are identical; pinned by `StringCharNeedlePredicateContractTests` + live), conditional (ternary) expressions, arithmetic (`+`/`-`/`*`/`/`/`%`), and string/date/math translations covered by provider tests. **String predicate semantics match .NET ordinal**: `Contains`/`StartsWith`/`EndsWith` are case-sensitive; an empty pattern matches every non-null row and never a NULL row; NEGATED string-method predicates keep NULL rows (a null string matches nothing, so the negation is true - the same C#-semantics rescue as negated comparisons); `string.IsNullOrEmpty` and `== null` behave like C#. Pinned by `StringPredicateSemanticsContractTests`. |
| `Select` entity and scalar projections | Supported | Fully translatable projections run server-side. |
| `Select` into anonymous types | Supported | Positional constructor matching against the projected columns. |
| `Select` into DTO with positional record | Supported | Constructor parameter names matched against projected member names. |
| `Select` into DTO with parameterized class constructor | Supported | Same constructor-match rule as records. |
| `Select` into DTO with init-only / writable property setters (`new T { A = x.A }`) | Supported | `MemberInit` projections bind each assignment to the DTO property; navigation-collection assignments are skipped. |
| `Select` with custom client logic | Constrained | Controlled by `DbContextOptions.ClientEvaluationPolicy`. The v1 default is `Throw`; `Warn` logs and allows the projection tail after server materialization, and `Allow` permits it silently. User-defined helper methods fall into this path. Strict provider mobility permits `Throw` and explicit `Warn`; silent `Allow` is compatibility-only. |
| `OrderBy`, `ThenBy` | Supported | Including the `Descending` variants. Provider-specific identifier escaping and expression translation apply. **String ordering follows the database column collation, not .NET `string.CompareTo` (CurrentCulture) semantics** - identical to EF Core; no SQL collation equals .NET CurrentCulture. On SQLite the default column collation is BINARY (ordinal), so uppercase A-Z sort before lowercase a-z. Order by a normalized / ordinal key for deterministic cross-provider ordering. Pinned by `StringOrderingContractTests`. **Guid ordering is consistent with .NET `Guid.CompareTo` on SQLite**: nORM stores a Guid as its canonical lowercase hex string and SQLite's BINARY collation orders that text ordinally, which - because each hex group is fixed-width big-endian and `Guid.CompareTo` compares the leading fields as unsigned - equals .NET's ordering for every Guid. (This depends on the TEXT storage; a byte/BLOB storage would diverge.) SQL Server orders `uniqueidentifier` by its own field order, a provider-native divergence under the same design exception as string collation. Pinned by `GuidOrderingContractTests`. **Floating-point ordering** (double/float) matches .NET for finite values and ±Infinity. Two provider-level edge behaviours on SQLite (identical to EF Core): NaN is not representable in a SQLite `REAL`, so storing it fails loud with `InvalidOperationException` ("Cannot store 'NaN' values.") rather than corrupting to NULL - NaN therefore never appears in an ordered set; and `-0.0` compares equal to `+0.0` in SQL (its sign is lost on read-back), whereas .NET `double.CompareTo` orders `-0` before `+0`, so add a tiebreak for deterministic zero placement. Pinned by `DoubleOrderingContractTests`. **Integer ordering** matches .NET for every integer CLR type whose range fits a signed 64-bit column - `int`/`long`/`short`/`sbyte` and `byte`/`uint`. `ulong` is stored as signed 64-bit (no provider has a portable native unsigned-64 type): values in `[0, long.MaxValue]` round-trip, order, compare, and compute exactly like a `long`; a value above `long.MaxValue` throws `NormUsageException` at write time on every path rather than wrapping to a negative bit pattern (the range contract is uniform across providers). Pinned by `IntegerOrderingContractTests` and `UlongStorageContractTests`. **Decimal ordering** matches .NET at full 28-digit precision on every provider. SQL Server / PostgreSQL / MySQL order the native `DECIMAL`; SQLite stores decimal as canonical text and orders it via a registered `NORM_DECIMAL` collation that compares with `decimal.Compare`, so magnitude, sign, trailing-zero equality, and differences beyond the 16th significant digit all order correctly (arithmetic and aggregation still use a `REAL` coercion on SQLite). Pinned by `DecimalOrderingContractTests`. **DateTime ordering** is chronological (matches `DateTime.CompareTo`) across sub-second (100ns) precision and the full Min/Max range; **DateTimeOffset ordering** is by UTC instant (matches `DateTimeOffset.CompareTo`), so mixed offsets sort by the moment they represent, not by wall-clock text. Pinned by `DateTimeOrderingContractTests`. **TimeSpan / bool / enum / char ordering** all match .NET: TimeSpan orders by value (multi-day and negative durations included, not by canonical-text lexical order), bool false&lt;true, enum by its underlying signed value (not declaration order), and char by code point. Pinned by `ScalarTypeOrderingContractTests`. **Value-converter columns** order by the STORED PROVIDER value (like EF Core): an order-preserving converter (e.g. a monotonic offset) orders the same as the CLR key, but an order-changing converter does not - notably an enum-to-string converter orders alphabetically by the stored name, not by the enum's underlying value. Order by an order-preserving stored representation for CLR-key ordering. Pinned by `ConverterColumnOrderingContractTests`. **Nullable ordering** matches .NET `Comparer<T?>.Default` (null is smallest: NULLs first ascending, last descending) on every provider. SQLite / SQL Server / MySQL do this natively; PostgreSQL (which defaults to NULLS LAST ascending) is normalized by prepending a same-direction `(key IS NOT NULL)` rank key, gated by the provider hook `RequiresExplicitNullOrderingForNullableKeys`. Pinned by `NullableOrderingContractTests`. |
| `Reverse` | Supported | Flips the active ORDER BY direction at the SQL layer. |
| `Skip`, `Take` | Supported | Provider-specific paging. `Skip(m).Take(n)` is the canonical pagination shape. `Take(n).Skip(m)` works with both literal counts (rewritten algebraically to `Skip(m).Take(n - m)`) and runtime parameters (emits `LIMIT (@take - @skip) OFFSET @skip` so the runtime values still produce the [skip, take) window). |
| `TakeLast`, `SkipLast` | Supported | Pages the tail by flipping the source ordering, paging the reversed sequence, and restoring result order after materialization. Requires an upstream `OrderBy`/`ThenBy` chain or a mapped key column for the default order. Composes with a directly preceding `Take`/`Skip` window: the windowed source wraps as a derived table and the tail counts from the end of the window, matching LINQ-to-Objects. |
| `Distinct` | Supported | Applies to the projected SQL shape. **Dedup matches .NET's default comparer**: decimal shapes dedup at full 28-digit precision via an exact scale-insensitive canonical key (17th-significant-digit values stay distinct; materialized values keep full precision), string shapes dedup case-sensitively (ordinal), and anonymous composites compare component-wise. The same exact keys apply to `Union`/`Intersect`/`Except` arms. Pinned by `DistinctSetOpSemanticsContractTests`. |
| `DistinctBy` | Supported | Keeps the first row per key in source order. nORM lowers this to `ROW_NUMBER() OVER (PARTITION BY key ORDER BY source-order) = 1`; use an explicit `OrderBy`/`ThenBy` chain when the chosen first row must be deterministic. |
| `DefaultIfEmpty` standalone | Supported | Applies after source materialization: non-empty sources are unchanged; empty sources return a single `null` element or the provided default value. Left-join `DefaultIfEmpty` is covered by the `GroupJoin` / `SelectMany` rows. |
| `Chunk` | Supported | The source query runs server-side unchanged; materialized rows are grouped into arrays of the requested size in result order, with a shorter final chunk when the row count is not a multiple. The size must be a constant or captured `int`; sizes below 1 throw `ArgumentOutOfRangeException`. |
| `Append`, `Prepend` | Supported | The element must be a constant or captured value — it is attached to the materialized rows in result order and never becomes part of the SQL. Row-derived elements throw `NormUnsupportedFeatureException`. |
| `Zip` | Supported | Second sequence may be a constant or captured local collection, or another database query; with or without a result selector, pairing truncates to the shorter side per LINQ semantics. Zipping two database queries requires an explicit `OrderBy`/`ThenBy` chain on BOTH sides (database row order is otherwise undefined, so positional pairing would be nondeterministic) and executes one round trip per side, pairing the materialized sequences in memory. |
| Operators after `Append`/`Prepend`/`Chunk`/`Zip`/`DefaultIfEmpty(value)` | Supported | The source query runs server-side; everything after the reshape is evaluated in memory with exact LINQ-to-Objects semantics. Scalar reducers (`Count`, `LongCount`, `Sum`, `Min`, `Max`, `Average`, `Any`, `All`, `Contains`), positional terminals (`First`, `Single`, `Last`, `ElementAt` and their `OrDefault` forms, `MinBy`, `MaxBy`), and sequence operators (`Where`, `Take`, `Skip`, `TakeLast`, `SkipLast`, `TakeWhile`, `SkipWhile`, `Distinct`, `Reverse`) all see the reshaped sequence — a prepended element shifts every index, an appended element is `Last`, and `Distinct` dedups with CLR equality. |
| `Count`, `LongCount`, `Any`, `All` | Supported | Includes predicate overloads and short-circuiting AND/OR. |
| Navigation aggregates: `parent.Children.Any(...)`, `.All(...)`, `.Count()`, `.LongCount()` | Supported | Emit correlated `EXISTS` / `NOT EXISTS` / scalar `(SELECT COUNT(*) ...)` subqueries against the dependent table. Two-hop chains (`parent.Children.SelectMany(c => c.GrandChildren).Count()`) emit a nested `IN` subquery. |
| Owned / many-to-many collection aggregates: `owner.Items.Count()`, `owner.Tags.Sum(t => t.Weight)` | Supported | `Count` / `LongCount` / `Any` and scalar `Sum` / `Min` / `Max` / `Average` (with a member selector) over an owned (`OwnsMany`) or many-to-many collection translate as correlated subqueries in both projections and predicates — the same shapes as ordinary relations, and composable under `OrderBy`. These collections are stored outside the relation graph, so before they were supported here such an aggregate silently returned a single wrong row; it now emits a correct per-owner subquery. A predicate lambda (`owner.Items.Count(x => x.Amount > 10)`) restricts the subquery, and for many-to-many the related entity's global / tenant filters also apply (the bridge table is joined to the related table, so a soft-deleted or other-tenant related row is excluded exactly as from the loaded collection). Single-key owner / link only; `All`, composite keys, and use under `AsOf` fail loud rather than returning a mismatched value. |
| Explicit-queryable correlated subqueries: `ctx.Query<X>()...` inside predicates and projections | Supported | No navigation property required. In predicates, `Any`/`All`/`Contains` lower to correlated `EXISTS` / `NOT EXISTS` / `IN`, and `Count`/`LongCount`/`Sum`/`Min`/`Max`/`Average` lower to correlated scalar subqueries comparable against outer columns or captured values. In projections, `Select(p => new { N = ctx.Query<Child>().Count(c => c.ParentId == p.Id) })` and the other scalar aggregates emit correlated scalar subquery columns. The subquery source may chain `Where` / `Select` / `OrderBy` / `ThenBy` / `Distinct` / `AsNoTracking` and may correlate to any enclosing query's row, including nested subqueries over the same entity type as the outer query. Value aggregates over an empty correlated set produce SQL `NULL` (bind a nullable result type); `Count` produces `0`. Subqueries that materialize rows (`ToList`, `First`, element access) or use operators outside that chain remain untranslatable and follow the client-evaluation policy. |
| `Sum`, `Average`, `Min`, `Max` | Supported | Both top-level aggregate queries and grouped aggregates, including computed selectors like `g.Sum(x => x.Price * x.Qty)` and conditional selectors like `g.Sum(x => x.Active ? x.Score : 0)`. **Decimal aggregates are exact at full decimal precision on every provider**: SQLite routes `Sum`/`Average` through registered decimal aggregate functions (decimal accumulation, not IEEE-754) and `Min`/`Max` through the `NORM_DECIMAL` collation so the extreme is the true one and always an actual stored value; server providers aggregate native `DECIMAL`. Decimal STORAGE round-trips the numeric value exactly at full 28-digit precision; the scale representation is normalized by the SQLite driver's canonical text (2m reads back as 2.0m - numerically equal, same as EF Core on SQLite; `Distinct`/equality merge scale variants exactly like C#). Pinned by `DecimalScaleRoundTripContractTests`. Computed selectors (e.g. `x.Price * x.Qty`) still evaluate the arithmetic itself in the provider's numeric type before aggregation. Empty-set and null semantics follow `Enumerable` (`Sum`=0, nullable `Average`/`Min`/`Max`=null, non-nullable throw); `Sum(int)` overflow fails loud, and an intermediate-only overflow with an in-range total returns the total (SQL wide-type accumulation, like EF Core). Pinned by `DecimalAggregatePrecisionContractTests` and `AggregateEdgeSemanticsContractTests`. |
| `GroupBy` | Supported | Single-column and composite anonymous-type keys (`p => new { p.A, p.B }`) supported. Aggregate projections (`g.Key`, `g.Count()`, `g.Sum(x => x.Amount)`, etc.) run server-side with SQL GROUP BY. Raw streaming (`GroupBy(key).ToListAsync()` → `List<IGrouping<K, V>>`) performs client-side grouping in memory after fetching all rows. Operators after a raw `GroupBy` apply to the grouped sequence: unpredicated `Count`/`LongCount` rewrite to fully-SQL `Select(key).Distinct().Count()` (one group per distinct key, including a null-key group) and unpredicated `Any` collapses to row existence; positional terminals, paging, and predicated quantifiers evaluate client-side over the assembled `IGrouping`s. **Key equality matches .NET's default comparer per type**: string keys group case-sensitively (ordinal), decimal keys group at full 28-digit precision (17th-significant-digit keys stay distinct), nullable keys produce a null group, converter-mapped keys group by the stored provider value and materialize back through the converter, and composite keys compare component-wise including null components. Pinned by `GroupByKeySemanticsContractTests`. |
| Inner joins | Supported | `Join` translates to provider SQL and is included in benchmark parity gates. Member-initialized DTO projections and computed projection members are covered by strict-mode and live-provider parity tests. **Join-key equality matches .NET's default comparer**: null keys never match null keys (both SQL and `Enumerable.Join` skip them; a `GroupJoin` outer row with a null key survives with an empty inner group), string keys join case-sensitively (ordinal), decimal keys join at full 28-digit precision, and composite keys compare component-wise with null components blocking the match. Pinned by `JoinKeySemanticsContractTests`. |
| Group joins | Supported | Simple key joins, bounded by `DbContextOptions.MaxGroupJoinSize`. Supports `AsAsyncEnumerable` streaming (groups are yielded one at a time). Terminals and sequence operators after a group-join projection (`First`, `Last`, `ElementAt`, `Skip`, `Take`, `TakeLast`, `SkipLast`, `Distinct`, `Reverse`, `Count`, `Sum`, `Any`, `All`, …) operate on whole groups, evaluated client-side over the assembled results — server-side paging on the flat joined rows would truncate a parent's children mid-group. The `GroupJoin` + `SelectMany` + `DefaultIfEmpty` (query-syntax left join) composition is fully supported, including projections that contain idiomatic null-guards (`c == null ? null : c.Prop` / `c != null ? c.Prop : null`) — the translator strips the redundant guard since LEFT JOIN already NULLs unmatched rows — and non-null fallback variants (`c == null ? "None" : c.Prop` / `c != null ? c.Prop : "None"`) which are lowered to `COALESCE(c.Prop, 'None')`. |
| `SelectMany` | Supported | Cross-product (`from a in A from b in B select new { a.X, b.Y }` → `CROSS JOIN`), navigation inner-join (with optional `Where` filter on the navigation), navigation `+ DefaultIfEmpty()` (→ `LEFT JOIN`), query-syntax left join (`GroupJoin + SelectMany + DefaultIfEmpty` → `LEFT JOIN`), and correlated expansion (`p => ctx.Query<Child>().Where(c => c.FK == p.PK)` → `INNER JOIN ... ON [predicate]`, including compound predicates). Result selectors (transparent identifier projections) work for all join forms. |
| Set operations: `Union`, `Intersect`, `Except`, `Concat` | Supported | `Concat` with a client-tail reshape (`Append`/`Prepend`/…) in either arm executes one round trip per arm and attaches the second arm's rows in memory; `Union`/`Intersect`/`Except` with a reshaped arm throw `NormUnsupportedFeatureException` (SQL dedups by column values, CLR by references — client evaluation would silently change semantics). Each side must project the same row shape. `Concat` lowers to `UNION ALL` under the same shape rule. `OrderBy` / `OrderByDescending` / `Skip` / `Take` AFTER the set op work (the trailing clauses apply to the unified result). `Where` AFTER the set op also works — the translator wraps the set-op SQL as `SELECT * FROM (<inner>) AS T0` so the predicate filters the unified rows. |
| Keyed set operators: `ExceptBy`, `IntersectBy`, `UnionBy` | Supported | Local key-set overloads for `ExceptBy`/`IntersectBy` lower to provider-side `IN`/`NOT IN` plus server-side `DistinctBy`. `UnionBy` with a local second entity sequence lowers the second sequence to a parameterized derived table, unions it with the source, then applies server-side row-number key dedupe with source rows ordered first. |
| `Include`, `ThenInclude` | Supported | Supported for mapped navigation paths including composite-key dependents, multi-level chains, and tenant-filtered contexts. Eager loading happens whenever `Include` is present; `AsSplitQuery()` is accepted for compatibility and adds no further effect. Unmapped navigation properties throw `NormUnsupportedFeatureException`. |
| `AsSplitQuery`, `AsNoTracking`, caching, temporal `AsOf` | Supported | These are nORM-specific query modifiers with dedicated tests. |
| Window functions: `WithRowNumber`, `WithRank`, `WithDenseRank`, `WithLag`, `WithLead` | Supported | nORM-specific query modifiers that project SQL window-function columns. Use an explicit `OrderBy` / `ThenBy` chain before the window call for deterministic results. **The OVER clause inherits the query ordering's type-exact contracts**: decimal order keys operate at full 28-digit precision (RANK ties only true duplicates), LAG/LEAD read true neighbours with the supplied default at window edges, and nullable orderings place nulls first like .NET. Pinned by `WindowFunctionKeySemanticsContractTests`. |
| Raw SQL composition | Provider-bound compatibility API | Two forms. **Terminal** — `FromSqlRawAsync<T>` / `FromSqlInterpolatedAsync<T>` execute a read-only `SELECT`/CTE immediately and return `List<T>`; the caller owns all filtering (no tenant predicate is added). **Composable** — `FromSqlRaw<T>(sql, args)` / `FromSqlInterpolated<T>($"...")` return an `IQueryable<T>` (implementing `INormQueryable<T>`, so `AsNoTracking`/`AsSplitQuery`/`Cacheable` and the async terminals chain directly off it): the SQL is wrapped as a derived table so `Where`, `OrderBy`, `Skip`/`Take`, projections, and the scalar terminals (`Count`, `Any`, `First`, `Sum`, …) layer on top, and **the entity's global (soft-delete) + tenant filters ARE applied to the outer query** — tenant is always enforced, the soft-delete filter is dropped by `IgnoreQueryFilters()` — so the SQL must `SELECT` the entity's columns (including the tenant column). Parameters bind positionally as `@p0`, `@p1`. Joins, `GroupBy`, `Include`, and set operators over a composable raw root fail loud rather than silently querying the mapped table. Neither form is provider-mobile generated surface; `UseStrictProviderMobility()` rejects raw SQL APIs before execution. |
| Dynamic table query `DbContext.Query(string)` | Provider-bound compatibility API | Reflects live schema into a runtime-generated CLR type. This is scaffolding/dynamic-table support, not provider-mobile generated application surface. Use typed `Query<T>()` for strict provider mobility. |
| Custom `[SqlFunction]` methods | Provider-bound compatibility API | User-authored SQL format strings are caller-owned dialect fragments. `UseStrictProviderMobility()` permits nORM-owned `NormFunctions` translations and rejects custom `[SqlFunction]` methods unless the app runs in compatibility mode with its own per-provider evidence. |
| `AsAsyncEnumerable` | Supported | Streams ordinary queries and GroupJoin queries (groups yielded one at a time). Include queries are rejected because eager-load paths require a coordinated multi-round-trip fetch incompatible with row-by-row streaming. |
| `ExecuteUpdateAsync` | Supported | Filtered and join-sourced entity queries, including composite-primary-key entities. Grouped and aggregated queries are rejected. Ordered and `Distinct` shapes run against the plain filter (neither changes an entity query's row set); paged and windowed shapes (`Take`/`Skip`/`TakeLast`/`SkipLast`, including operators composed after the window) resolve their target rows through a keyed subquery over the full query SQL and require key columns on the entity. Join sources: single-PK → `WHERE pk IN (SELECT T0.pk ...)`, composite-PK → row-tuple `WHERE (pk1, pk2) IN (SELECT T0.pk1, T0.pk2 ...)` (SQLite/Postgres/MySQL) or JOIN-based form (SQL Server). Two assignment forms: (1) literal / captured-local value — `SetProperty(x => x.Foo, 42)`; (2) server-side computed expression — `SetProperty(x => x.Counter, x => x.Counter + 1)` supporting member access, constants, arithmetic (`+`/`-`/`*`/`/`/`%`), and Convert. |
| `ExecuteDeleteAsync` | Supported | Filtered and join-sourced entity queries, including composite-primary-key entities and correlated `EXISTS`/`NOT EXISTS` via `Where(p => !ctx.Query<Other>().Any(o => o.K == p.K))`. Grouped and aggregated queries are rejected. Ordered and `Distinct` shapes run against the plain filter (neither changes an entity query's row set); paged and windowed shapes (`Take`/`Skip`/`TakeLast`/`SkipLast`, including operators composed after the window) resolve their target rows through a keyed subquery over the full query SQL and require key columns on the entity. Join sources: single-PK → `WHERE pk IN (SELECT T0.pk ...)`, composite-PK → row-tuple `WHERE (pk1, pk2) IN (SELECT T0.pk1, T0.pk2 ...)` (SQLite/Postgres/MySQL) or JOIN-based form (SQL Server). |
| Arbitrary CLR methods in predicates | Unsupported | Unsupported methods/members throw from expression translation instead of being evaluated client-side in filters. |
| Arbitrary client evaluation before server filtering/paging | Unsupported | Server filters and paging must happen before any allowed client projection tail. |
| `TakeWhile` / `SkipWhile` | Constrained | Supported for an explicit `OrderBy` / `ThenBy` source with the one-argument or index-aware predicate overload. nORM lowers this to a provider-mobile row-numbered cumulative break flag with zero-based LINQ index semantics. Composes with a directly preceding `Take`/`Skip` window: the windowed source wraps as a derived table (its ORDER BY + paging clause stays inside the wrap) and the break flag evaluates the window's rows in window order, with the index counting within the window. Unordered forms throw deterministically rather than buffering the entire table client-side. |
| `OfType<T>` / `Cast<T>` | Constrained | Identity pass-through when `T` equals the source element type, or (for reference types) is a base class assignable from it. `OfType<TDerived>()` is also supported for TPH hierarchies: when `TDerived` carries `[DiscriminatorValue]` and its base type carries `[DiscriminatorColumn]`, the translator injects the discriminator `WHERE` predicate automatically — composing with further `Where(...)` calls works as expected. Downcasting to a type with no discriminator metadata still throws `NormUnsupportedFeatureException`. |
| `SequenceEqual` | Constrained | Supported when provider queryable sources have explicit `OrderBy` / `ThenBy` chains and the default equality comparer. nORM row-numbers queryable sources and compares `(row_number, columns)` bidirectionally with provider SQL. A local in-memory second sequence of mapped entities is parameterized as an ordered derived table and compared server-side. Unordered queryable sources and custom comparer overloads throw deterministically. |

## Value translations

| Feature | Status | Notes |
| --- | --- | --- |
| `string` methods: `ToUpper`, `ToLower`, `Length`, `Trim`/`TrimStart`/`TrimEnd`, `Substring`, `Replace`, `IndexOf`, `Contains`, `StartsWith`, `EndsWith` | Supported | Provider-specific syntax. `IndexOf(value, StringComparison)` supports literal/captured comparison modes by using provider case-sensitive wrappers or lower-case normalization. `Replace(old, new, StringComparison)` supports literal/captured case-sensitive modes; ignore-case replacement fails deterministically because native `REPLACE` semantics are not provider-mobile. |
| `string` statics: `IsNullOrEmpty`, `IsNullOrWhiteSpace`, `Concat`, `Compare`, `CompareOrdinal`, `CompareTo` | Supported | `Compare` / `CompareTo` emit a `CASE` returning `-1` / `0` / `1`. `string.Compare(a, b, StringComparison)` and `string.Compare(a, b, bool ignoreCase)` are supported when the comparison mode is a literal or captured value; ignore-case modes lower both sides through provider SQL. `CompareOrdinal` uses provider case-sensitive wrappers. |
| `string.Format` / interpolated strings | Supported | Constant templates with positional placeholders lower to provider concatenation. Fixed numeric specs (`F<N>`), common date/time tokens (`yyyy`, `yy`, `MM`, `dd`, `HH`, `mm`, `ss`), and alignment are translated through provider hooks. Locale-aware/custom .NET formatting outside that subset remains client-projection work. |
| `Regex.IsMatch` | Constrained | PostgreSQL and MySQL use native regex primitives. SQLite emits `REGEXP` backed by deterministic nORM-registered managed functions. SQL Server translates a simple ASCII-safe subset to `LIKE` / `LEN` / `RIGHT` predicates (`^` / `$` anchors, literal text, simple bracket classes such as `[A-Z]`, `\d`, `\w`, and anchored class-plus-fixed-suffix shapes like `^[A-Z]+\d{2}$`) and throws `NormUnsupportedFeatureException` for complex regex constructs instead of drifting semantics. |
| `Regex.Replace` | Constrained | PostgreSQL and MySQL use native regex-replace primitives. SQLite emits `regexp_replace` backed by deterministic nORM-registered managed functions. SQL Server translates the literal-pattern/literal-replacement subset to `REPLACE` with explicit collation and throws `NormUnsupportedFeatureException` for regex constructs, captures, or replacement substitutions. |
| `Convert.ToInt32` / `ToInt64` / `ToString` / `ToBoolean` / `ToDouble` / `ToDecimal` / etc. | Supported | Emit a provider `CAST`. |
| `Math.Abs`, `Ceiling`, `Floor`, `Round`, `Sqrt`, `Pow`, `Exp`, `Log`, `Log10`, `Sign`, `Min`, `Max`, `Truncate` | Supported | Per-provider mapping (`LEAST`/`GREATEST` on PG/MySQL, `CASE` for SQL Server `Min`/`Max`, etc.). |
| `DateTime` members: `Year`, `Month`, `Day`, `Hour`, `Minute`, `Second`, `DayOfYear`, `DayOfWeek`, `Date` | Supported | Provider-specific extraction; `DayOfWeek` normalized to `System.DayOfWeek` (`0`=Sun..`6`=Sat) on every provider. |
| `DateTime.AddDays` / `AddMonths` / `AddYears` / `AddHours` / `AddMinutes` / `AddSeconds` | Supported | Provider-specific interval arithmetic. |
| `DateTime.UtcNow` / `DateTime.Now` / `DateTime.Today` in predicates | Supported | Evaluated as a bound parameter at execution time (static-member constant path). |
| `DateTimeOffset` members | Supported | Same member surface as `DateTime`; materializer parses SQLite TEXT into `DateTimeOffset`. |
| `DateOnly.Year` / `Month` / `Day` / `DayOfYear` | Supported | Per provider. |
| `TimeOnly.Hour` / `Minute` / `Second` | Supported | Per provider. |

| `Convert.ChangeType(col, typeof(T))` or captured `Type` value | Supported | Pattern-matched at query-build time; emits `CAST(col AS X)` for int/short/byte/sbyte/long/string/double/float/decimal/bool. Non-constant target-type expressions remain unsupported because SQL cannot dispatch per row on CLR `Type`. |
| `DateTimeOffset.LocalDateTime` accessor (projection / WHERE / OrderBy) | Supported | Per-instant local timezone semantics for UTC instants in the documented 1900-2100 range. nORM computes `TimeZoneInfo.Local` offset ranges at query-build time and lowers them to provider SQL `CASE` expressions, avoiding provider-specific timezone databases while preserving DST-aware .NET `LocalDateTime` behavior. Values outside the generated range use the current local offset fallback. |
| `DateTimeOffset` col `==` / `!=` DateTime literal | Constrained | Lowered to UTC-instant equality via per-provider epoch-microsecond hooks. Microsecond-resolution: full .NET 100ns tick equality is not portable across SQLite TEXT/date functions and native provider precisions. `.Kind == Utc` literals get offset 0; Unspecified / Local literals use `TimeZoneInfo.Local.GetUtcOffset(literal)` for the literal instant. |

| LINQ `Aggregate` sum-fold (1-arg + seed forms) | Supported | `(acc, x) => acc + x` lowers to `Sum(selector)` server-side. 2-arg overload adds the seed in CLR after the SQL sum. 1-arg form throws on empty source per .NET semantics. |
| LINQ `Aggregate` min/max-fold (Math.Max/Min + Conditional shapes) | Supported | Recognised shapes: `Math.Max(acc, x)`, `Math.Min(acc, x)`, `(acc, x) => x > acc ? x : acc` (and mirrored / `>=` / `<` / `<=` variants). | LINQ `Aggregate` string-concat fold (simple + seed-aware separator) | Supported | `(acc, x) => acc + sep + x` and `(acc, x) => acc + (acc == "" ? "" : sep) + x` both lower to `GroupBy(_ => 1).Select(g => string.Join(sep, g.Select(...)))` so the provider's `STRING_AGG` / `GROUP_CONCAT` runs on the server. Empty source returns seed (3-arg) or throws (1-arg). |
| `Enum.TryParse<T>(stringCol, out T)` as WHERE predicate | Supported | Lowers to a case-sensitive provider comparison against enum names; the 3-arg `ignoreCase` overload lowers through `LOWER(col) IN (...)` when `ignoreCase` is a constant or captured bool. The richer "TryParse + use out-var in same predicate" shape is not a parseability check; use `col == nameof(X)` instead. |

| Arithmetic operators (`+`, `-`, `*`, `/`, `%`) in `Where`, `Select`, and aggregate selectors | Supported | Column types drive numeric semantics. |

| Local-collection `Contains` (`ids.Contains(x.Id)`) | Supported | Null-aware: emits `(col IN (...) OR col IS NULL)` when the list contains nulls. |
| `Guid.Empty` and other static-field constants in predicates | Supported | Evaluated at execution time via the static-member constant path. |
| `Guid.NewGuid()` in queries | Supported | Translates to the provider-owned server GUID primitive (`NEWID()`, `gen_random_uuid()`, `UUID()`, or canonical SQLite random GUID text). ## Provider Notes

The matrix is provider-neutral unless a provider-specific note says otherwise.
The same query shape should either work on every supported provider or fail with
a documented provider limitation. Provider-specific syntax differences are
handled by `DatabaseProvider` implementations and cross-provider tests.

## Documentation Rules

- README wording must point here instead of claiming "full LINQ" or "complete
  LINQ" without qualification.
- New LINQ translations need a test that proves SQL shape or result behavior.
- New unsupported LINQ shapes need a test that proves the exception is stable
  and does not fall back to unsafe client-side filtering.
- `docs/linq-support-coverage.md` maps each non-unsupported row in this matrix
  to executable test files and is enforced by documentation contract tests.
